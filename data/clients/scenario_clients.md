# Сценарные клиенты

В БД `clients.sqlite` 35 клиентов с фиксированными ID `C-000001 — C-000035` имеют гарантированно заданные параметры. Они используются в Q&A для проверки tools+RAG логики на конкретных профилях. Остальные ~350 клиентов сгенерированы случайно по реалистичным распределениям и не привязаны к конкретным Q&A.

Для воспроизводимости БД пересоздаётся скриптом `data/generation/01_generate_clients.py` с фиксированным seed; ID клиентов и их параметры стабильны.

## Принципы покрытия

Каждый сценарный клиент покрыт минимум одним Q&A; ключевые профили (для эскалации, расчётов, краевых случаев) — несколькими (до 10).

## Привязка клиентов в Q&A

В Q&A 127 из 180 кейсов имеют `client_id`, 53 — `null`. Логика привязки:

- **Авторизованные каналы (`chat_intern`, `mobile`)** — `client_id` обязателен по регламенту (п. 2.3.1 РП-ОБ-005: при авторизации в интернет-банке/мобильном приложении клиент идентифицирован). Даже если данные клиента не нужны для ответа (например, общий вопрос «какие у вас кредиты?»), привязка сохраняется — это контекст обращения. Используются как сценарные клиенты (там, где это контекстно осмысленно — например, вопрос про зарплатный проект задаёт C-000014), так и случайные из БД для разнообразия.
- **Анонимные каналы (`chat_site`, `contact_center`)** — `client_id` отсутствует. Это новые/неидентифицированные обращения; клиент в системе ещё не существует.

Случайные клиенты (`C-000100+`) используются как контекст для общих вопросов на авторизованных каналах. Их финансовые показатели не разбираются в `expected_behavior`, но модель должна корректно работать с этим контекстом (например, не использовать данные клиента там, где они не нужны для ответа).

## Что не покрыто и почему

- Большинство сценарных клиентов используются для проверки конкретных правил регламента (просрочки, закрытые отрасли, стоп-факторы). Для расширения датасета можно добавить новые сценарные профили под новые правила.
- Несколько профилей умышленно сделаны редкими (банкротство, самозанятый <12 мес НПД, бизнес 8 мес), чтобы Q&A корректно проверял отказы.
