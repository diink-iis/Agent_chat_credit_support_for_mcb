# Категории Q&A датасета

Датасет покрывает E2E-сценарии работы автоматизированного помощника поддержки кредитования МСБ. Каждый кейс размечен категорией и подкатегорией для аналитики качества.

## Структура записи

Каждая строка `qa.jsonl` — один кейс со следующими полями:

| Поле | Описание |
|---|---|
| `id` | Уникальный идентификатор Q-NNN |
| `category` | Основная категория (см. ниже) |
| `subcategory` | Уточнение внутри категории |
| `channel` | `chat_intern` (авторизован) / `chat_site` (анонимный) / `mobile` / `contact_center` |
| `client_id` | ID клиента из `clients.sqlite` или `null` для анонимных запросов |
| `history` | Массив предыдущих сообщений в диалоге для multi-turn (пустой для single-turn) |
| `question` | Текущая реплика клиента |
| `expected_behavior` | Что должен сделать Помощник |
| `expected_outcome_type` | `info` / `calculation` / `escalation` / `rejection` / `clarification` |
| `referenced_documents` | Массив пунктов нормативки, которые нужны для ответа |
| `difficulty` | `easy` / `medium` / `hard` |
| `is_multiturn` | true/false |

## Категории и распределение

| Код | Название | Кол-во | Доля |
|---|---|---|---|
| `info` | Информационные запросы | 45 | 25% |
| `transactional` | Транзакционные с привязкой к клиенту | 45 | 25% |
| `escalation_sales` | Эскалация: явное намерение оформить продукт | 18 | 10% |
| `escalation_negative` | Эскалация: негатив клиента | 18 | 10% |
| `edge_no_data` | Краевой: нет данных в нормативке | 18 | 10% |
| `edge_conflict` | Краевой: коллизия в документах | 9 | 5% |
| `edge_manipulation` | Краевой: манипуляции / попытки обхода | 18 | 10% |
| `offtopic` | Не по теме | 9 | 5% |
| **Итого** | | **180** | 100% |

Multi-turn кейсы составляют ~25% (45 кейсов), распределены по всем категориям.

## Подкатегории

### info
- `info_products` — про продукты, их условия и параметры
- `info_requirements` — про требования к заёмщику
- `info_documents` — про необходимые документы
- `info_application_process` — про процесс подачи и каналы
- `info_early_repayment` — про правила досрочного погашения
- `info_restructuring` — про правила реструктуризации
- `info_sla_status` — про сроки рассмотрения и статусы
- `info_general` — общие вопросы про сегменты, тарифы, действие регламентов

### transactional
- `tx_application_status` — статус конкретной заявки клиента
- `tx_loan_state` — состояние действующего кредита
- `tx_repayment_calc` — расчёт суммы досрочного погашения
- `tx_eligible_products` — какие продукты доступны клиенту
- `tx_loan_terms` — условия конкретного договора клиента

### escalation_sales
Все триггеры из п. 4.2 РП-ОБ-005. Подкатегории — по типу намерения:
- `sales_new_loan` — намерение оформить новый кредит
- `sales_open_account` — намерение открыть расчётный счёт
- `sales_restructuring_init` — намерение инициировать реструктуризацию
- `sales_early_repayment_init` — намерение оформить ДП (не информационный запрос)
- `sales_consultation` — намерение получить персональную консультацию по подбору

### escalation_negative
- `neg_complaint_threat` — угрозы жалобами в надзорные органы / суд / СМИ
- `neg_offensive` — оскорбительная или агрессивная лексика
- `neg_request_human` — прямая просьба переключить на человека
- `neg_long_unresolved` — длительно нерешённая проблема, эмоциональный накал
- `neg_distress` — обращение, указывающее на тяжёлое состояние клиента

### edge_no_data
- `nodata_product_outside_scope` — продукт за пределами сегмента/линейки
- `nodata_specific_terms` — специфические условия не в регламенте
- `nodata_other_segment` — вопросы для иного сегмента (средний бизнес)
- `nodata_other_products` — вопросы про не-кредитные продукты Банка
- `nodata_other_orgs` — вопросы про другие организации/банки

### edge_conflict
Коллизии, специально заложенные в документах:
- `conflict_account_term` — общий vs Бизнес-Старт срок счёта
- `conflict_debt_load` — общий vs Бизнес-Развитие долговая нагрузка
- `conflict_seasonal_revenue` — общая 12 мес vs сезонная 24 мес выручка
- `conflict_payment_priority` — приоритет планового платежа vs ДП в один день
- `conflict_product_vs_general` — иные приоритеты продуктового над общим

### edge_manipulation
- `manip_request_exception` — просьба сделать исключение из регламента
- `manip_third_party_data` — попытка получить данные третьего лица
- `manip_false_status` — ложные заявления о статусе клиента
- `manip_prompt_injection` — попытки переопределить инструкции
- `manip_provoke_promise` — попытки выпросить гарантии и обещания

### offtopic
- `offtopic_general` — посторонние темы (погода, личные)
- `offtopic_other_finance` — общие финансовые вопросы (инвестиции, налоги)
- `offtopic_political_sensitive` — политические/чувствительные темы

## Принципы оценки

Поле `expected_outcome_type` задаёт тип финального результата, по которому в первую очередь оценивается ответ:

- `info` — информация подана корректно и достаточно полно;
- `calculation` — расчёт выполнен по верной формуле и с верными исходными;
- `escalation` — эскалация выполнена с корректным триггером и переданным контекстом;
- `rejection` — корректный отказ с честным объяснением (не просьба обратиться куда-то «на всякий случай», а именно отказ с обоснованием);
- `clarification` — запрошено уточнение, без которого ответ некорректен.

Поле `expected_behavior` содержит развёрнутое описание того, что именно ожидается от Помощника, включая ограничения (например: «не раскрывать конкретные критерии скоринга», «не обещать одобрение»).
