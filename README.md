# Учебный кейс: Автоматизация поддержки кредитования МСБ

Синтетический датасет для учебного кейса E2E. Студенты разрабатывают агента на LangGraph + GigaChat, который обслуживает обращения клиентов малого и микробизнеса по вопросам кредитования.

Решение должно подключать промпты, RAG (по нормативным документам), инструменты (доступ к БД клиентов) и корректно эскалировать обращения на оператора.

## Структура репозитория

```
data/
  documents/         # 5 нормативных документов в markdown — RAG база
  clients/           # БД клиентов (SQLite) + карта сценарных клиентов
  qa/                # Q&A датасет для E2E-оценки + описание категорий
  generation/        # Скрипты воспроизводимой генерации синтетики (для преподавателя)
README.md
```

## Документы (RAG)

Пять нормативных документов «Банка» — реалистичные регламенты, по которым работает Помощник:

1. [01_credit_products.md](data/documents/01_credit_products.md) — продуктовая линейка (5 кредитных продуктов), требования к заёмщикам, скоринговая модель, тарифы.
2. [02_application_process.md](data/documents/02_application_process.md) — каналы подачи, перечни документов, SLA, статусы заявок, дозапросы, отказы, повторная подача.
3. [03_early_repayment.md](data/documents/03_early_repayment.md) — правила досрочного погашения по каждому продукту, расчёт сумм, особые случаи.
4. [04_restructuring.md](data/documents/04_restructuring.md) — основания и виды реструктуризации, ограничения, процесс, последствия для клиента.
5. [05_customer_communication.md](data/documents/05_customer_communication.md) — регламент работы Помощника с обращениями, **триггеры эскалации** (продажи и негатив), компетенция, конфиденциальность.

В документах сознательно заложены:
- Контролируемые коллизии общего и продуктового регламента (приоритет имеет продуктовый) — например, общий срок счёта 6 мес vs Бизнес-Старт 3 мес; общий лимит долговой нагрузки 50% vs Бизнес-Развитие 70%.
- Исключения для сезонных отраслей (строительство, с/х) — расчёт от 24-мес среднегодовой выручки.
- Ссылки между документами по номерам (`№ РП-КР-001`, пункты `2.4.4` и т.п.) — для проверки качества чанкования и поиска.
- Запреты на раскрытие критериев скоринга — для проверки корректных отказов от детализации.

## База клиентов (tools)

[clients.sqlite](data/clients/clients.sqlite) — 385 клиентов (35 сценарных + 350 случайных), 86 действующих кредитов, 143 заявки.

Таблицы:
- `clients` — клиент, его форма (ООО/ИП/Самозанятый), отрасль, выручка, скоринг, отношения с Банком (счёт, зарплатный проект), история просрочек.
- `credit_products` — действующие кредитные договоры с остатками, графиками, статусом просрочки, обеспечением.
- `applications` — история заявок (включая активные) со статусами и решениями.

Сценарные клиенты `C-000001 — C-000035` имеют гарантированно заданные профили и используются в Q&A. Полная карта — в [scenario_clients.md](data/clients/scenario_clients.md).

Схема БД: [schema.sql](data/generation/schema.sql).

## Q&A датасет (E2E-оценка)

[qa.jsonl](data/qa/qa.jsonl) — 180 кейсов, размеченных категориями. Формат и подкатегории описаны в [categories.md](data/qa/categories.md).

Распределение:

| Категория | Кол-во | Доля | Что проверяет |
|---|---|---|---|
| `info` | 45 | 25% | Информационные ответы по продуктам, требованиям, документам, процессам — основная работа RAG |
| `transactional` | 45 | 25% | Запросы с привязкой к конкретному клиенту — RAG + tools |
| `escalation_sales` | 18 | 10% | Триггер эскалации «намерение оформить продукт» |
| `escalation_negative` | 18 | 10% | Триггер эскалации «негатив клиента» |
| `edge_no_data` | 18 | 10% | Корректные отказы по вопросам вне нормативки |
| `edge_conflict` | 9 | 5% | Коллизии общего vs продуктового регламента |
| `edge_manipulation` | 18 | 10% | Манипуляции, попытки получить чужие данные, prompt injection |
| `offtopic` | 9 | 5% | Корректное закрытие нерелевантных тем |

34 кейса (19%) — multi-turn диалоги. Распределены по всем категориям там, где органичны (нарастание негатива, уточнения, попытки переформулировать манипуляцию).

**Привязка к клиентам.** 127 из 180 кейсов имеют `client_id` — это все обращения через авторизованные каналы (`chat_intern`, `mobile`). На таких каналах клиент идентифицирован по факту авторизации (п. 2.3.1 РП-ОБ-005), даже если его данные не нужны для ответа. 53 кейса без `client_id` — анонимные обращения через `chat_site` или `contact_center`. См. подробности в [scenario_clients.md](data/clients/scenario_clients.md).

Поле `expected_outcome_type` задаёт тип финального результата: `info` / `calculation` / `escalation` / `rejection` / `clarification`. Поле `referenced_documents` — точные ссылки на пункты нормативки, нужные для корректного ответа.

## Воспроизводимость

Вся синтетика воспроизводима:

```bash
# Перегенерировать БД клиентов из спецификаций (фиксированный seed):
python3 data/generation/01_generate_clients.py

# Прогнать валидацию Q&A (структура, ссылки, согласованность с БД):
python3 data/generation/02_validate_qa.py
```

Q&A составлен вручную с привязкой к конкретным сценарным клиентам и пунктам документов — не генерируется автоматически.

Скрипты предназначены для преподавателя. Студентам передаются только итоговые артефакты (`data/documents/`, `data/clients/clients.sqlite`, `data/qa/qa.jsonl`); скрипты генерации и `scenario_clients.md` им показывать не нужно — это разметка ответов.

## Что использовать на каком этапе

| Этап работы студента | Артефакты |
|---|---|
| Построение RAG | `data/documents/` |
| Построение tools | `data/clients/clients.sqlite` + схема (`data/generation/schema.sql`) |
| Тестирование агента в разработке | свои тестовые кейсы или подмножество `qa.jsonl` |
| Финальная E2E-оценка | весь `qa.jsonl` с метриками по категориям |

Метрики оценки рекомендуется считать **по категориям отдельно**, так как разные категории проверяют разные аспекты:
- по `info` и `transactional` — точность ответа;
- по `escalation_*` — корректность срабатывания триггера и переданного контекста;
- по `edge_*` — корректность отказа, без выдумывания и без обхода ограничений;
- по `offtopic` — вежливое отклонение без потери лица системы.

Под RAG и tools отдельно генерируются специализированные датасеты оценки качества (вне scope этого репозитория).
