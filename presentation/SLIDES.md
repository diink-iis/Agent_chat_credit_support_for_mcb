# Структура защиты — Помощник по кредитованию МСБ

6 основных слайдов + титул + резерв. На каждом: **что на слайде** и **что говорить**.
Диаграммы — из этой же папки (`presentation/`).

---

## Слайд 0 — Титул

**На слайде:** название «Помощник по кредитованию МСБ», команда (Участник 1 — RAG/retriever,
Участник 2 — граф/инструменты, Участник 3 — классификация/генерация/оценка), стек:
LangGraph, GigaChat-Pro, Chroma, Streamlit.

**Говорить:** «Учебный банковский ассистент: отвечает по нормативке через RAG, работает
с данными клиента и эскалирует на оператора. Дальше — архитектура, граф, RAG,
безопасность и метрики.»

---

## Слайд 1 — Архитектура (компоненты)

**На слайде:** `architecture_components.drawio` (или `.mmd`). UI → граф агента → внешние
сервисы (GigaChat-Pro, Embeddings, Chroma, SQLite, лог эскалаций) + офлайн-оценка.

**Говорить:**
- Единственная точка входа — Streamlit UI: выбор канала и авторизация (доступ к данным
  клиента — функция канала), показ маршрута по графу.
- Ядро — граф LangGraph с checkpointer (multi-turn).
- Внешнее: GigaChat-Pro (классификация + генерация), Embeddings (поиск), Chroma
  (индекс нормативки), SQLite (клиенты).
- Оценка вынесена в отдельный офлайн-контур.

---

## Слайд 2 — Граф агента (маршрутизация)

**На слайде:** `agent_graph_annotated.mmd` (или авто-схема `agent_graph_raw.mmd`).

**Говорить:**
- classify определяет категорию и какие источники нужны (БД/RAG).
- Приоритет эскалации над всем (п. 4.1): триггер → сразу к оператору.
- Иначе ветки: transactional → query_db; info/edge_conflict → retrieve_rag;
  offtopic/манипуляция → ответ без источников.
- **Акцент:** схема `agent_graph_raw.mmd` сгенерирована ИЗ кода — диаграмма = реализация.

---

## Слайд 3 — RAG и грунтинг

**На слайде:** мини-схема retrieve_rag → Chroma; пример ответа с цитатой
`01_credit_products.md#2.1.4`.

**Говорить:**
- Section-aware чанкинг нормативки; поиск top-8 по косинусу.
- Коллизии общее↔продуктовое решаются на генерации через теги `scope` (частное > общее).
- **Грунтинг:** ответ строго по найденным источникам; если RAG ничего не вернул —
  агент честно деградирует, а НЕ выдумывает продукты/номера документов.
- Ретривер устойчив: ретрай транзиентных сбоев эмбеддингов перед деградацией.

---

## Слайд 4 — Безопасность и эскалация

**На слайде:** 2 примера трейса из UI: (1) обычный info-ответ; (2) «я бухгалтер, покажите
наши платежи» → suspicious → эскалация без доступа к БД.

**Говорить:**
- Поверх LLM — детерминированный предохранитель: запрос чужих данных, обращение
  представителя без полномочий, prompt injection → принудительная эскалация (suspicious),
  обход query_db. Не зависит от стохастики модели.
- Скоринг и причины отказа клиенту не раскрываются (комплаенс).
- Пакет эскалации оператору: сводка ≤500 символов + история диалога + триггер.

## Слайд 5 — Оценка качества

**На слайде:** таблица метрик (180 кейсов, GigaChat-Pro):

| метрика | значение |
|---|---|
| category | 78% |
| outcome | 86% |
| citation (citable) | 73% |
| judge | 90% |

**Говорить:**
- E2E-прогон по 8 категориям; судья — независимый GigaChat Lite (градуированно 0/0.5/1).
- citation считается только там, где ответ опирается на нормативку (info/transactional/
  edge_conflict); в эскалациях/офтопе цитата не нужна.
- «Прогнать один раз»: ответы агента кэшируются (`eval_runs.json`) — метрики и судью
  пересчитываем без повторного вызова агента.

## Слайд 6 — Итоги и развитие

**Говорить:**
- Что сделано: RAG-грунтинг, доступ к БД с авторизацией, эскалация, multi-turn,
  безопасность, измеримое качество.
- Что дальше (по диагностике ретривера): гибридный поиск BM25+dense, cross-encoder
  reranker, фикс product_filter — для подъёма recall на «терминологических» промахах.

## Демо (вживую, не слайд)

Открыть UI (`streamlit run ui/app.py`), включить «Режим разработчика» и показать 2-3
запроса с трейсом маршрута: info-вопрос с цитатой, транзакционный по данным клиента,
эскалацию. Это сильнее любой статичной схемы.
