# E2E-оценка агента (Участник 3, задачи 3.5–3.6)

Скрипт [`evaluate.py`](evaluate.py) прогоняет все 180 кейсов `data/qa/qa.jsonl`
через собранный граф и считает метрики **отдельно по 8 категориям** (как
рекомендует корневой README: разные категории проверяют разные аспекты).

## Метрики

| Метрика | Что показывает | Откуда gold |
|---|---|---|
| `category` | точность классификатора (узел `classify`) | `qa.category` |
| `outcome` | совпал ли тип финального результата | `qa.expected_outcome_type` |
| `escalation` | для `escalation_*`: эскалировано + верный триггер | категория |
| `citation` | gold-источник попал в источники ответа. Считается **только** для citable-категорий (`info`, `transactional`, `edge_conflict`), где ответ опирается на нормативку; в refusal/escalation/offtopic корректный ответ цитаты не содержит → метрика неприменима (`—`) | `qa.referenced_documents` |
| `judge` | LLM-судья на GigaChat: ответ выполняет `expected_behavior` (опц., `--judge`) | `qa.expected_behavior` |

### Модели

- **База агента (classify + generate): `GigaChat-Pro`** — выше потолок на пограничных
  `intent ↔ info` кейсах. Задаётся в `agent/llm.make_gigachat_deps` (переопределяется
  `GIGACHAT_MODEL`).
- **Судья: `GigaChat` (Lite)** — дешевле и независим от модели агента (меньше риск
  «само себя хвалит»). Задаётся явно в `evaluate.make_judge`.
- Эмбеддинги RAG: `Embeddings` (GigaChat, 1024 dims), `rag/embeddings.py`.

### Прогон один раз (кэш ответов)

Дорогой проход агента (RAG + Pro) отделён от оценки и сохраняется в `eval_runs.json`,
чтобы метрики и судью можно было пересчитывать **без повторных вызовов агента**:

```bash
python evaluate.py --mode gigachat        # 1 раз: агент + RAG, кэш ответов → eval_runs.json
python evaluate.py --from-runs --judge     # затем: только судья (Lite) поверх кэша
python evaluate.py --from-runs             # пересчёт метрик с кэша (бесплатно)
```

### Методическая заметка по citation

В `qa.jsonl` `referenced_documents` проставлен у **всех** 180 кейсов, включая
`offtopic`/`edge_manipulation`/`escalation_*`/`edge_no_data`. Но правильное поведение
агента в этих категориях — эскалация / отказ / вежливое перенаправление **без**
цитаты нормативки, поэтому подсчёт citation там штрафовал бы за корректное поведение
(что подтверждает judge 75–100%). С учётом этого citation считается только по
citable-категориям; осмысленный показатель грунтинга ≈ взвешенное по `info` /
`transactional` / `edge_conflict`, а не размытый общий по 8 категориям.

## Как запускать

```bash
# Оффлайн baseline (без GigaChat): правила + FakeRetriever + шаблон.
.venv/bin/python evaluate.py --mode stub

# Боевой режим: реальный Retriever Участника 1 + GigaChat (нужен .env + rag/index).
.venv/bin/python evaluate.py --mode gigachat
.venv/bin/python evaluate.py --mode gigachat --judge          # + LLM-судья
.venv/bin/python evaluate.py --mode gigachat --categories info,edge_conflict --limit 30
```

Детальные пер-кейсные результаты пишутся в `eval_results.json`.

## Подготовка к сдаче: корректность, безопасность, полнота

По итогам аудита реальных ответов (judge-прогон) устранены дефекты, заметные при
приёмке:

1. **Безопасность — детерминированный предохранитель (`agent/nodes.py`,
   `_security_override`).** Аудит выявил утечку: «Я бухгалтер ООО …, какие у нас
   платежи?» в авторизованной сессии классифицировался как self-query и **выдавал
   данные**. Поверх LLM добавлен высокоточный детектор (запрос чужих данных,
   обращение представителя без полномочий п. 8.6, prompt injection п. 7.3): такие
   паттерны ВСЕГДА уводятся в `suspicious`-эскалацию и не доходят до `query_db` —
   независимо от стохастики модели. Покрыто тестом `test_layer3.test_security_override`
   (срабатывает на угрозе, не задевает легитимный self-query).
2. **Граница intent ↔ доступность/условия (п. 4.2.1/4.2.3).** «Хочу закрыть кредит,
   какие условия?», «через сколько можно подать после отказа», «какие продукты мне
   доступны» больше не эскалируются как намерение (обрабатываются сами); при этом
   «мне нужен оборотный 5 млн», «подберите под задачу», «открыть счёт ради кредита»
   корректно остаются `intent`. `transactional` category 56%→78%.
3. **Полнота транзакционных ответов.** Генератор использует читаемую сводку данных
   клиента (а не игнорирует её) и отвечает конкретными значениями; `tx_eligible_products`
   закрыт детерминированным `get_eligible_products`; добавлен стоп-фактор банкротства
   (п. 3.6). `transactional` citation 44%→69%.
4. **Корректность цитат.** Аудит 65 ссылок на пункты нормативки: 97% валидны (ни
   одного систематического выдумывания пунктов).
5. **Multi-turn: контекст диалога в генераторе** (`agent/llm.py`). Раньше генератор
   получал только последнюю реплику (`latest_client_text`), поэтому уточнения
   («а если внесу 1 млн?», «а как разблокировать?») теряли продукт/тему из прошлых
   ходов — агент отвечал невпопад (галлюцинация «нельзя для нескольких кредитов»,
   общее «обратитесь в отделение»). Теперь подаётся история диалога с пометкой, что
   уточнения относятся к теме выше. На целевых кейсах (Q-079/080/087) ответы стали
   корректными; multi-turn judge был 74% против 87% single-turn.
6. **Устойчивость к сбоям** (`agent/nodes.py`, `evaluate.py`). `retrieve_rag` ловит
   сбой поиска/эмбеддингов и деградирует (пустой контекст вместо падения, п. 4.4.3);
   evaluator переживает падение отдельного кейса. Проверено на инциденте: при волне
   из 121 транзиентного сбоя GigaChat прогон **завершился** (раньше первый же сбой
   ронял весь прогон из 180).

> **Замечание по измерению.** Финальный чистый прогон выполнен на `GigaChat-Pro`
> (баланс эмбеддингов пополнен, HTTP 402 более не воспроизводится): RAG отрабатывает,
> `citation` считается. Заголовочная таблица — с этого прогона; ответы агента
> закэшированы в `eval_runs.json`, судья пересчитывается поверх кэша без повторного
> вызова агента (`python evaluate.py --from-runs --judge`).

## Дожатие качества (по результатам judge-прогонов)

Серия итераций по найденным в прогонах причинам:

1. **Формат данных клиента + закрытие утечки скоринга** (`agent/llm.py`). Генератор
   игнорировал сырой JSON `tool_results` и отправлял «смотрите в интернет-банке».
   Введена читаемая сводка (`_format_tool_results`: профиль, кредиты с остатком/
   ставкой/сроком/осталось мес/просрочкой, заявки), поданная ПЕРЕД нормативкой как
   авторитетный источник. Поля скоринга (`credit_score`, `current_debt_load`,
   `decision_reason_category`) **полностью исключены** из контекста — раньше модель
   раскрывала рейтинг клиенту (нарушение п. 6.2).
2. **Инструмент `get_eligible_products`** (`agent/tools.py`, вызывается в `query_db`
   по маркерам доступности). Детерминированный подбор продуктов по порогам РП-КР-001
   (выручка, срок деятельности, срок счёта, сегмент, отрасль) + доступ по рейтингу
   (п. 4.1) — но **значение скоринга не возвращается**, только вердикт и нескоринговые
   причины. Закрыл нерешаемую ранее подкатегорию `tx_eligible_products`.
3. **Few-shot hard negatives** в `classify.md` для границы transactional ↔ intent/info
   (повторная подача, «какие МНЕ доступны», «подойдёт ли мне X» vs «в принципе»).
4. **top_k 5→8** (выше потолок citation), **temperature генератора → 0.0**
   (детерминизм), **градуированный судья 0/0.5/1** (меньше шум измерения).

Итог против исходного боевого baseline:
`transactional` category **56%→71%**, citation **44%→62%**, outcome 64%→69%; общий
`citation` 37%→41%. Утечка скоринга устранена. Остаток ошибок `transactional` —
строгость судьи (часть «провалов» верны по сути) и неустранимая граница info↔tx.

## Итерация 3.6: что было сделано по результатам прогона

1. **Детектор социнженерии (п. 7.3).** Классификатор (и промпт, и baseline)
   разделяет `edge_manipulation` на два исхода: запрос чужих данных / переопределение
   инструкций → эскалация с триггером `suspicious`; выпрашивание исключения/гарантии
   → отказ. Это подняло `edge_manipulation` в stub: category 6%→50%, outcome 17%→61%.
   Соответственно обновлён ожидаемый исход в `tests/test_layer3.py` (манипуляция-кейс
   теперь `escalation`, как в `qa.jsonl`).
2. **Расширены маркеры транзакционных запросов** (статус заявки, состояние кредита,
   «мне отказали/одобрили», «следующий платёж») — `transactional` category 4%→22%.

## Калибровка `outcome_type` (применено по результатам прогона)

Прогон обнажил рассогласование эвристики `outcome_type` в `agent/nodes.py` с
таксономией датасета — главный потолок метрики `outcome`. Исправлено:

- **`transactional` → почти всегда `info` в gold** (29/45), `calculation` лишь 4.
  Узел ставил `calculation` всякий раз при `needs_db=true`, из-за чего *улучшение*
  классификатора (4%→22%) *снижало* outcome (60%→47%). Теперь `calculation`
  ставится только при маркерах расчёта ДП (`_is_repayment_calc`), иначе `info`.
  Итог: `transactional` outcome 47%→**64%**.
- **`offtopic` → `info`** (9/9) и **`edge_no_data` → `info`** (17/18): убраны из
  `_REJECTION_CATEGORIES` (там остался только `edge_manipulation`). Вежливое
  перенаправление и честное «нет данных» в датасете считаются `info`, а `rejection`
  зарезервирован за отказом от недопустимого. Итог: `offtopic` outcome 78%→**100%**.

Затронуты узел `generate_answer` (`agent/nodes.py`) и ожидания в `tests/test_layer3.py`
(transactional→info, offtopic→info, манипуляция→escalation). Общий `outcome` 64%→**70%**.
