# Промпт-шаблон: разрешение коллизий

Вставляется в системный/инструктивный промпт генерации **поверх** контекста из RAG.

## Зачем

Retriever возвращает релевантные чанки по смыслу запроса (top-k по score) и **не**
переставляет их принудительно. Когда по одному вопросу в контекст попадают и общее
правило (`scope: general`), и продуктовое исключение (`scope: specific`), модель
должна сама выбрать частный случай. Каждый чанк уже снабжён тегами `scope`,
`product_code`, `section_id`, `doc_id` — опирайся на них.

## Правило приоритета (вставлять в промпт генерации)

```
ПРАВИЛО РАЗРЕШЕНИЯ ПРОТИВОРЕЧИЙ В РЕГЛАМЕНТАХ:

Тебе дан контекст из нескольких пунктов нормативных документов. У каждого пункта
есть тег области действия:
  • scope = "specific" — правило конкретного кредитного продукта (исключение);
  • scope = "general"  — общее правило линейки МСБ.

Если общий и продуктовый пункт противоречат друг другу по одному и тому же вопросу:
  1. ПРИОРИТЕТ имеет пункт с scope = "specific" для того продукта, о котором речь.
     Продуктовое правило — это сознательное исключение из общего, а не ошибка.
  2. Общее правило (scope = "general") применяется только когда продуктового
     исключения нет.
  3. Не «усредняй» и не выбирай произвольно: явно назови, какое правило общее, какое
     — продуктовое исключение, и какое применяется в данном случае.
  4. Всегда ссылайся на источник в формате «п. <section_id> <Документ №>».
     Если коллизию объясняешь — приведи ОБА пункта (общий и частный).
  5. Никогда не выдумывай числа и условия, которых нет в контексте. Если нужного
     пункта в контексте нет — скажи об этом и не отвечай по памяти.
```

## Эталонные примеры коллизий (из qa.jsonl, категория edge_conflict)

| Вопрос | Общее правило (general) | Продуктовое исключение (specific) | Что отвечать |
|--------|-------------------------|-----------------------------------|--------------|
| Срок ведения счёта для Бизнес-Старт | п. 3.5 — счёт от 6 мес | п. 2.4.3 — Бизнес-Старт допускает от 3 мес | Применяется исключение: 3 мес |
| Долговая нагрузка по инвест-кредиту | п. 3.4 — не более 50% выручки | п. 2.2.5 — Бизнес-Развитие при покрытии ≥130% до 70% | Применяется исключение: до 70% |
| Окно досрочки через интернет-банк | п. 4.2.1 — до 18:00 МСК | п. 4.2 — исключение для Бизнес-Развитие | Назвать общее окно + исключение продукта |
| Срок рассмотрения заявки | «до 30 дней» из иных сегментов | п. 4.2 — для МСБ действуют SLA из таблицы | Для МСБ — только SLA таблицы п. 4.2 |

## Как это работает в связке

1. `retrieve(query, k=5)` → список чанков с тегами `scope`/`product_code`/`section_id`.
2. Чанки передаются в промпт генерации **вместе с тегами** (не только текст).
3. Промпт выше заставляет модель выбрать `specific` над `general` при конфликте и
   сослаться на оба пункта.

Коллизия разрешается на уровне генерации, а не ранжирования — так retriever сохраняет
высокий recall на общих регламентах (эскалации, процессы), а правильность выбора
частного правила обеспечивает LLM по тегам.
